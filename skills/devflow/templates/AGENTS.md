## DevFlow Workflow

This project uses DevFlow for structured development.

### Quick Start

> **Important: If `.devflow/` already exists in this project, use it directly. Do NOT run `devflow init`.**

```bash
# Check for existing workflows first
devflow list-workflows

# Select a workflow
devflow select-workflow <workflow-id>

# Get current instruction
devflow current

# Mark done and advance (or auto-routed on failure)
devflow done
```

### How It Works

1. **Progressive Disclosure**: DevFlow shows one step at a time via CLI
2. **Gate System**: Each step has conditions that must be satisfied
3. **Fail Routes**: On gate failure, engine auto-routes based on failure count (e.g., 3 fails → escalate)
4. **Cross-Workflow**: Steps can transition to other workflows (e.g., MODE-B debug → MODE-A review)
5. **Workflow Files**: Defined in `.devflow/workflows/*.toml`
6. **Config Variables**: Gates can reference `{test_command}`, `{lint_command}` etc.

### Key Concepts

- **Gates** must pass before advancing. If they fail, the engine may auto-route you to a different step.
- **Fail count persists** across visits — repeated failures escalate to different routes (e.g., 1-2 fails → retry, 3+ fails → ask human).
- **Cross-workflow transitions** happen automatically (e.g., after MODE-B debug-finish, you enter MODE-A at write-plan).
- Do NOT skip steps or manually change workflow state. Only `devflow done` advances the workflow.

See `.devflow/workflows/` for available workflows in this project.
