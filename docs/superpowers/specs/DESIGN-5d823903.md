# DESIGN-5d823903: Ralph Loop Integration Architecture

## Context

We need to add an autonomous Loop Engine on top of DevFlow's existing `WorkflowEngine`. The loop should drive the workflow step-by-step without requiring the agent to manually run `devflow current` -> act -> `devflow done`. A critical constraint: **checkpoints must not directly commit to VCS**; instead they generate diff/patch files.

## Approach A: In-Process Local Loop (MVP)

**Description**: `LoopEngine` runs entirely inside the current Python process. It iterates over the backlog, builds a prompt, and calls an internal `AgentRunner` class that executes the step logic directly (reading files, writing code, running shell commands).

**Pros**:
- Fastest to implement and test
- No external CLI dependencies
- Easy to debug with standard Python tooling
- Full access to DevFlow internals

**Cons**:
- No "fresh context" benefit — the same process accumulates state
- If the agent makes a mistake, it may compound across iterations
- Less aligned with Ralph Loop's core philosophy of clean-slate iterations

**Effort**: 1-2 days

## Approach B: Subprocess Agent Runner (Recommended)

**Description**: `LoopEngine` spawns a dedicated Python subprocess for each iteration via `subprocess.run([sys.executable, "-m", "devflow.agent_runner"])`. The subprocess receives the prompt via **stdin**, executes the step in isolation, and exits. `LoopEngine` then calls `engine.advance()` to validate gates.

**Pros**:
- True process isolation per iteration (fresh memory, no context pollution)
- Can be extended later to invoke external CLIs (`claude`, `amp`)
- Checkpoint diff files are generated between iterations, giving clean boundaries
- Still uses Python standard library, no extra install requirements

**Cons**:
- Slightly more overhead than in-process
- Need to design `agent_runner` CLI interface and prompt protocol
- Subprocess stdout/stderr parsing for error detection

**Effort**: 2-3 days

## Approach C: External CLI Bridge

**Description**: `LoopEngine` detects and invokes external AI CLIs (`claude --print`, `amp --non-interactive`, `kimi --print`) as subprocesses. The prompt instructs the external CLI to complete one step and exit.

**Pros**:
- True fresh AI context every iteration (the holy grail of Ralph Loop)
- Can leverage powerful external models
- Maximum decoupling between Loop Engine and agent logic

**Cons**:
- Requires users to install external CLIs
- Different CLIs have incompatible flags and prompt passing mechanisms
- Non-deterministic availability and version drift
- Harder to test in CI

**Effort**: 3-5 days

## Recommendation

**Choose Approach B (Subprocess Agent Runner)** for the following reasons:

1. **Architecture balance**: It gives us process-level isolation (fresh memory per step) without the fragility of depending on third-party CLI tools.
2. **Incremental path**: Once Approach B is solid, adding external CLI support (Approach C) is a matter of swapping the spawn command in `_spawn_agent()`.
3. **Checkpoint alignment**: Since we generate diff/patch files instead of committing, having a clean subprocess boundary makes it easy to capture the exact state change produced by each iteration.
4. **Testability**: We can write pytest integration tests that run the full loop end-to-end without requiring users to install `claude` or `kimi`.

## Key Design Decisions

### 1. Checkpoint = Diff File, Not Commit

- **Git**: run `git diff` before/after each iteration, save output to `.devflow/checkpoints/CHECKPOINT-<run_id>-<step_id>.diff`
- **SVN**: run `svn diff`, save to `.devflow/checkpoints/CHECKPOINT-<run_id>-<step_id>.patch`
- **No VCS**: snapshot modified files (based on mtime or a simple file list) into `.devflow/checkpoints/CHECKPOINT-<run_id>-<step_id>/`

### 2. Backlog Auto-Generation

`devflow sync-backlog` reads the current workflow TOML and produces `.devflow/backlog.json` with one task per step. This makes the Loop Engine agnostic to how the backlog was created.

### 3. Progress as Append-Only Log

`.devflow/progress.md` stores:
- Timestamp
- Step ID
- Checkpoint file path
- Any errors or learnings

Before each iteration, `LoopEngine` injects the last 3-5 entries into the agent prompt.

### 4. Step-Level Loop Behavior

Workflow TOML will support optional `loop_behavior` per step:
- `"run"`: proceed normally
- `"pause"`: stop loop, ask for human input (e.g., `user_approved` gates)
- `"skip"`: skip during loop mode (useful for docs-only steps in hotfixes)

### 5. Windows Compatibility for Subprocess Agent Runner

On Windows, `_spawn_agent()` must handle three platform-specific concerns:

1. **Prompt via stdin (not argv)**: Windows `CreateProcessW` has ~32KB command-line limit and poor escaping for multi-line prompts. The prompt is passed via `input=prompt` to `subprocess.run()`, and `agent_runner` reads from `sys.stdin`.

2. **`creationflags=subprocess.CREATE_NO_WINDOW`**: Prevents console window flashes when the subprocess launches (critical when running inside GUI editors like VS Code). Applied conditionally via `hasattr(subprocess, "CREATE_NO_WINDOW")`.

3. **Explicit `encoding="utf-8"`**: Windows default console encoding is often `cp936` or `cp1252`. Both `subprocess.run(..., text=True, encoding="utf-8", errors="replace")` and `agent_runner` stdout wrapping are required.

Example implementation:
```python
kwargs = {
    "cwd": project_root,
    "capture_output": True,
    "text": True,
    "encoding": "utf-8",
    "errors": "replace",
    "timeout": 300,
    "input": prompt,
}
if hasattr(subprocess, "CREATE_NO_WINDOW"):
    kwargs["creationflags"] = subprocess.CREATE_NO_WINDOW
result = subprocess.run([sys.executable, "-m", "devflow.agent_runner"], **kwargs)
```
